# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A SAP CAP (Node.js) project working toward an **AI Procurement Assistant**: turn procurement requirements scattered across email, PDF, images, Excel, and partner REST feeds into a human-reviewed Sourcing Project and, on approval, a Purchase Requisition in SAP S/4HANA Cloud. AI proposes (extraction, normalization, duplicate detection, Material Group/Commodity/Supplier recommendation, project drafting); a human always approves — the AI never creates SAP business objects itself.

The full target design — business architecture, domain model, CDS entities, AI/RAG architecture, S/4HANA integration, security, roadmap — is in **[docs/solution-architecture.md](docs/solution-architecture.md)**. Read it before making architectural decisions; this section only summarizes where the code currently stands relative to it.

**Current state (Phase 0 baseline, not yet the target domain)**: what's actually implemented today is a generic document-upload + RAG-chat scaffold called "Document & Chat Hub" (`poc.sp.hub`) — upload a document, chunk it, embed the chunks, chat with it via retrieval-augmented Q&A. It has no procurement/sourcing domain logic; treat it as the technical template for the target `IntakeService` (upload → chunk → status pattern), not as existing business functionality.

Much of the AI/RAG logic is not yet implemented: `srv/lib/embedder.js` and `srv/lib/vector_search.js` are empty stub files, and the Fiori chat UI extension (`ChatSection.js`) is generator-scaffolded placeholder code (a button wired to a `MessageToast`). The CDS data model and the document CRUD/status/delete actions in `srv/document-service.js` are the only fully working parts.

Per the roadmap in `docs/solution-architecture.md` §29, upcoming work replaces the `workspace` namespace with the `sourcing` domain model (§10) and adds `IntakeService`/`WorkspaceService`/`SourcingProjectService`/`KnowledgeService` alongside — the existing `DocumentService` is kept as the intake pattern template, not deleted.

**Sourcing domain — built so far**: `db/sourcing-schema.cds` (namespace `sourcing`, the full §10 model) plus four CAP services, each `.cds` + `.js`:

- `IntakeService` (`/api/intake`) — `uploadDocument(workspaceId, originType, fileName, fileType, content)`, `extractRequirements`, `getExtractionStatus`. `SourceDocument.content` (`LargeString`) holds raw text for the fully-implemented parsers (email/CSV-TSV/REST); binary formats (PDF bytes, `.xlsx`, images) can be uploaded as metadata but fail extraction with an informative Phase-2 message until their parsers support binary content.
- `WorkspaceService` (`/api/workspace`) — CRUD over `RequirementWorkspaces`/`WorkspaceRequirements` + `merge`/`split`/`reject`/`regenerate`/`promoteToSourcingProject`. A `before UPDATE` hook auto-marks edited rows `aiStatus='EDITED'`; deletes and mutations are written to `AuditLog`.
- `SourcingProjectService` (`/api/sourcing`) — CRUD over the project + its composition children and read-only master data + `generateDraft`/`approve`/`submitToS4`.
- `KnowledgeService` (`/api/knowledge`) — `@requires: 'KnowledgeCurator'`; `indexDocument`/`reindex`/`listByCategory` over the curated `KnowledgeDocument` corpus.

**Convention for not-yet-built AI/integration steps**: operations that depend on the Phase 5 S/4HANA call are implemented as deliberate stubs — they validate inputs and existence first, then `req.reject(501, '... not implemented yet (Phase N)')`. This currently covers only `SourcingProjectService.submitToS4`. Guardrail gates that don't need AI are real and enforced now (e.g. `submitToS4` requires status `APPROVED`; `promoteToSourcingProject` blocks unreviewed low-confidence items). Every other action that was a stub (`KnowledgeService.reindex`, `SourcingProjectService.generateDraft`, `WorkspaceService.regenerate`, `IntakeService.extractRequirements`) is now wired to the real AI modules (see below).

**Dev auth**: `package.json` `cds.requires.auth` now configures `kind: mocked` with three test users — `curator` (KnowledgeCurator), `requester` (ProcurementRequester), `manager` (ProcurementRequester + ProcurementManager). Use these for HTTP Basic auth in local testing (e.g. `curl -u curator: ...`); production still uses `xsuaa`. `KnowledgeService` restricts only its **write actions** (`indexDocument`, `reindex`) to `KnowledgeCurator` — reads of the curated corpus are open, which also lets the CAP Fiori preview / read clients work without a browser login. (A service-level `@requires` blocks anonymous `$metadata` XHRs and makes the Fiori preview fail with "Failed to load UI5 component"; that's why the restriction sits on the actions, not the service. To harden reads for production, add `@requires: 'authenticated-user'` at the service level and use the browser basic-auth login for the preview.) The other services are open in dev.

**AI module (`srv/ai/`) — in progress**: every AI capability goes through one adapter, `srv/ai/llm-client.js` (§14) — `chat({system, user, schema, ...})` returns a schema-validated object (single retry on invalid JSON, §16); `embed(text)` returns a `Vector(3072)`-compatible array. It selects a **provider** from config/env (`AI_PROVIDER`): `openai` (native `fetch`, no SDK dependency, needs `OPENAI_API_KEY`) or `mock` (deterministic, offline — **the default**, so dev/CI run with no key and no network). Guardrails baked in (§25): output-token cap, input-size guard, clear failure when the OpenAI key is missing. Config knobs via `cds.env.ai` or env vars (`AI_CHAT_MODEL`, `AI_EMBED_MODEL`, etc.). The module includes a tiny built-in JSON-schema validator (exported as `.validate`) — no `ajv` dependency.

**Enabling real AI (mock → openai)**: the **`mock` provider does not read input** — `synthesize()` fills every schema field with `mock-<random>` / the first enum value, deterministically. So under mock, "10 laptops / 10 keyboards" yields *one* placeholder requirement (`mock-3553`, qty `0.770`), not two structured rows — this is expected offline behavior, not a pipeline bug (the parse → extract → `WorkspaceRequirement` + traceable `RequirementSource` plumbing is all correct; only the *values* are synthetic). To get real structured extraction: copy **`.env.example` → `.env`** (gitignored) and set `AI_PROVIDER=openai` + `OPENAI_API_KEY`, then restart. **`.env` loading**: `@sap/cds` does **not** read `.env` on its own (verified) — the project's **`srv/server.js`** bootstrap loads it via Node's built-in `process.loadEnvFile` (no `dotenv` dep) before delegating to `@sap/cds/server`, so it works for any launch method (`cds watch` / `cds serve` / `npm start`). Missing `.env` is fine (mock default). A shell-exported var overrides `.env`. Env is not hot-reloaded — restart after edits. **Security**: `.env` is gitignored; `.env.example` is the committed template and must never hold a real key. **Embed model dims**: keep `text-embedding-3-large` (3072) to match the `Vector(3072)` column — `text-embedding-3-small` is 1536 and would mismatch unless the column + `embeddingDimensions` config change together. Defaults: `gpt-4o-mini` chat, `text-embedding-3-large` embed (3072 dims — **must** stay 3072 to match the `Vector(3072)` column). `OPENAI_BASE_URL` can point at an OpenAI-compatible endpoint (Azure OpenAI, SAP Generative AI Hub, a proxy). Verified the full openai path end-to-end against a **local fake OpenAI endpoint** and then against the **real OpenAI API**: `AI_PROVIDER=openai` extracts real structured requirements from document text (e.g. an email → `inverted microscopes with phase contrast`×2 due 2026-09-15, `nitrile gloves size M`×10); and `openai` with **no/invalid key** fails cleanly (`401`/"OpenAI API key is not configured") → surfaces as `SourceDocument.status='FAILED'` with that errorMsg, never a silent empty result.

**Two real-OpenAI bugs found + fixed during first live extraction** (mock never exercised them): (1) **`json_object` 400** — OpenAI rejects `response_format: json_object` unless the literal word "json" appears in the messages; `openaiChat` now guarantees it. (2) **wrong field names** — without the schema in the prompt, the model invents its own keys (`items` instead of `requirements`, `deliveryDate` instead of `requestedDate`, `textSnippet` instead of `rawSnippet`), so validation fails with `$.requirements: required`. Fix: `openaiChat` now **embeds the JSON schema** in the system prompt ("use these exact field names") — the mock path is unaffected (it synthesizes from the schema directly). Also hardened `extraction.js` `normalizeRequirement`: LLMs sometimes emit the literal string `"null"`/`"none"`/`"n/a"` for an absent unit/snippet — `cleanString()` maps those to real `null`.

**Embedding model access + dimensions** (found when Regenerate/enrichment 403'd — `Project ... does not have access to model 'text-embedding-3-large'`): the RAG path (`WorkspaceService.regenerate` → `enrichment` → `embedder`) needs a working **embed** model, and many OpenAI projects only grant `text-embedding-3-small`. Switched the default embed model to **`text-embedding-3-small`** — but its max output is **1536 dims** (it can't produce 3072), so the `KnowledgeDocument.embedding` column moved from `Vector(3072)` → **`Vector(1536)`** and `embeddingDimensions` config default → 1536, to match. **These three must always agree**: `AI_EMBED_MODEL`, `embeddingDimensions`, and the Vector column size — change them together. (`-large`/3072 needs both project access to that model AND `Vector(3072)`.) The legacy chat-hub `db/schema.cds` still declares `Vector(3072)` — untouched, it's the separate `workspace`-namespace example, not the sourcing RAG path. After changing dims, run `KnowledgeService.reindex` once to re-embed the corpus. Verified live: reindex embeds 5 docs, and `regenerate` succeeds on multiple requirements with real RAG grounding (`enriched "Microscope, 40x objective": 1 MG, 1 CC, 1 suppliers`, resolving `MG-LAB-001` from the real corpus).

Built on the adapter so far:

- `srv/ai/extraction.js` — `extractRequirements(text)` → `{ requirements: [...] }` (each with description/quantity/unit/requestedDate/rawSnippet/confidence), temperature 0.1, strict schema, defensive normalization (confidence clamped 0..1, dates → ISO or null). Pure logic — no DB/OData knowledge; the service maps its output to `WorkspaceRequirement`/`RequirementSource` rows (§18).
- `srv/ai/document-parsers/` — registry (`parseDocument({originType, fileType, buffer, text})`) → `{ text, segments }`, where each segment has a `location` (subject / paragraph N / row N / page N) for `RequirementSource` traceability (§17). **Plain-Text/Email/REST/CSV-TSV are fully implemented dependency-free**; PDF lazily requires `pdf-parse`, `.xlsx` needs the `xlsx` package, and image OCR throws a Phase-2 message — all with informative errors, not silent failures. **`originType` enum includes `Text`** (mapped to the passthrough `text-parser`, same as `RestApi`) — the correct route for pasted/plain content, and the **default** option in the Intake Hub upload dialog. **Email-parser robustness fix**: the `.eml` parser splits headers from body on the first blank line (`\n\n`); a headerless plain-text upload (e.g. a pasted shopping list with no blank line) used to be parsed *entirely as headers*, leaving the body empty and extraction failing with "non-empty document text" (this was a real FAILED-status bug on plain content uploaded as `Email`). Now `looksLikeHeaders()` guards this — the leading block is only treated as headers when its first non-empty line matches `Name: value`; otherwise the whole input is the body. So even content mis-typed as `Email` still extracts. Real emails (with a subject header + blank line) are unaffected.
- `srv/ai/embedder.js` — embedding + RAG retrieval over the curated `KnowledgeDocument` corpus (§15). `embed(text)`, `indexDocument(id)`, `reindex({onlyMissing})`, and `search(queryText, {category, topK, minScore})`. Note: on the dev **sqlite** driver a `Vector(3072)` column round-trips as a **JSON string**, so read-back is parsed and cosine similarity is computed in JS (correct + portable at PoC corpus size; on HANA prod, push `COSINE_SIMILARITY` into SQL for scale). `search` returns top-K with no score floor by default; pass `minScore` to threshold. This is the file that will make `KnowledgeService.reindex` real and ground `enrichment`/`project-drafting`. **Not the same as** the empty legacy `srv/lib/embedder.js` (chat-scaffold stub).

- `srv/ai/duplicate-detection.js` — `detectDuplicates(requirements, {threshold})` (§3 step 4, §18). Combines three explainable signals — semantic cosine (via embedder), lexical Jaccard of description tokens, structural unit/quantity match — and returns `{ pairs, assignments }`: scored pairs with human-readable `reasons`, plus union-find `assignments` proposing each duplicate's canonical `duplicateOf` (lowest index in the group). Only proposes; merging stays a human act (`WorkspaceService.merge`, §25). Pure logic, no DB. The three-signal blend degrades gracefully — lexical+structural still catch near-dups even when semantic embeddings are unavailable (as under the mock provider).

- `srv/ai/enrichment.js` — `enrichRequirement(requirement, {topK, maxPerType})` (§5, §15, §16). RAG-grounded: one `embedder.search` per type (MaterialGroupCatalog / CommodityTaxonomy / SupplierProfile), then `llm-client.chat` (temp 0.3, strict schema) returns ranked `materialGroups`/`commodityCodes`/`suppliers`, each with clamped confidence and a `citation` to the knowledge `sourceRef`. Also returns `grounding` (the retrieved sourceRefs per type) for auditability. Proposes only (§25). This is what `WorkspaceService.regenerate` will call.

- `srv/ai/project-drafting.js` — `draftSourcingProject(requirements, {topK, maxRisks, workspaceTitle})` (§20, §16). RAG-grounded in `Guideline` + `PastProject` knowledge, `llm-client.chat` (temp 0.5, strict named-field schema) proposes title/description/category/priority (`Low|Medium|High`) / timeline (ISO or null) and a `risks` list (each with a `Low|Medium|High|Critical` severity + mitigation), plus `grounding` sourceRefs. Proposes only; the service marks persisted rows `aiGenerated=true`. This is what `SourcingProjectService.generateDraft` will call.

**All pure-AI modules under `srv/ai/` are now built and validated.** Date normalization is strict on purpose: `new Date()` leniently accepts junk (e.g. `"mock-42"` → year 2041, `"mock-213161"` → year 213161), so `toIsoDateOrNull` (in `extraction.js` and `project-drafting.js`) requires an ISO leading date and a plausible year (1970–2100) before trusting a value. Note: `clamp01`, `toIsoDateOrNull`, and the RAG `formatContext` helper are currently duplicated across a few AI files — a candidate for extraction into a shared `srv/ai/util.js`.

**Wiring progress**:

- `KnowledgeService.reindex(onlyMissing?)` is real — it calls `embedder.reindex()` and returns `{ documentsQueued }`; seeded `KnowledgeDocument` rows have no usable vector until this runs once (verified: `embedder.search` returns 0 hits before, correct hits after).
- `SourcingProjectService.generateDraft` is real — it calls `project-drafting.draftSourcingProject` with the project's `Requirement` rows, writes the narrative fields (title/description/category/priority/timeline) onto `SourcingProject`, and replaces only the **AI-authored** `Risk` rows (`aiGenerated=true`) so a human-added risk survives repeated regeneration. Gated to `status='DRAFT'` (409 once approved — the project is frozen, §20/§25) and requires at least one `Requirement` (400 otherwise). Writes a `GENERATE_DRAFT` audit entry with `aiInvolved=true` (verified via direct SQL against the DB file).
- `WorkspaceService.regenerate` is real — it calls `enrichment.enrichRequirement` and takes the top-ranked Material Group / Commodity Code candidate, but **only assigns the code if it resolves against real `MaterialGroup`/`CommodityCode` master data** (`SELECT ... WHERE code = <ai-code>`) — an AI-hallucinated code is never written as a dangling reference (§25), since these associations have no DB-level FK constraint to catch it. Resets `aiStatus` to `PROPOSED` for re-review regardless of prior status. Writes a `REGENERATE` audit entry with `aiInvolved=true` and the retrieval `grounding` (verified: with a real seeded corpus, grounding correctly cites the real `KnowledgeDocument.sourceRef`s even though the mock's generated _values_ are nonsense — proving retrieval and generation are independently correct).

- `IntakeService.extractRequirements` is real — `parseDocument()` (§17) then `extraction.extractRequirements()` (§18) turn `SourceDocument.content` into `WorkspaceRequirement`/`RequirementSource` rows (`aiStatus='PROPOSED'`); a `RequirementSource.location` is filled in by a best-effort match of the requirement's `rawSnippet` against the parser's segments, falling back to `"document"`. Status machine: `UPLOADED → EXTRACTING → EXTRACTED` (blocks re-extraction, 409) or `→ FAILED` with `errorMsg` (retry allowed). **Important CAP lesson learned here**: a parse/extraction failure is handled by persisting `FAILED`+`errorMsg` and returning normally (HTTP 200, `{status:'FAILED', itemsCreated:0}`) rather than via `req.reject()` — calling `req.reject()` rolls back the _entire_ request's transaction, which would silently undo the `FAILED` status write made earlier in the same handler (confirmed via a direct repro before fixing).

**All four originally-stubbed AI/RAG actions are now wired end-to-end.** Only `SourcingProjectService.submitToS4` (Phase 5, S/4HANA OData) remains a stub. `uploadDocument` now also takes `workspaceId` and `content` params — a `SourceDocument` must belong to a `RequirementWorkspace` (400/404 if missing/invalid) and needs `content` before extraction can run (400 otherwise). The seeded `test/data/sourcing-SourceDocument.csv` rows predate the `content` field and have none — they're fine as read-only demo data (their `status` is already set) but calling `extractRequirements` on them will hit the "no content" guard.

**Validation harnesses** for the AI files live in the session scratchpad (not committed): they run each module offline against the mock provider. There is no committed test runner yet.

## Architecture

**Service boundary**: single CAP service, `DocumentService`, mounted at `/api/documents` (`srv/document-service.cds` + `srv/document-service.js`). The `Documents` entity is exposed as a projection (excluding the `chunks` composition); chunks, chat sessions, and messages are managed internally through custom actions/functions, not exposed as separate service entities.

**Data model** (`db/schema.cds`, namespace `workspace`):

- `Documents` → composition of many `DocumentChunks` (chunking + embedding status lives on the parent: `status`, `chunkCount`, `errorMsg`).
- `DocumentChunks.embedding` is a `Vector(3072)` column (OpenAI `text-embedding-3-large` dimensionality) for similarity search — this is what `srv/lib/vector_search.js` and `srv/lib/embedder.js` are meant to populate/query once implemented.
- `ChatSessions` optionally link to a `Documents` record and composition-own many `ChatMessages` (each message has a `role`, `content`, and an optional `sources` field for citing retrieved chunks).
- Cascading delete is handled manually in application code (`deleteDocument` action in `document-service.js`), not via DB-level cascade: it deletes messages → sessions → chunks → the document, in that order, because compositions alone don't cascade across the session/document link.

**Persistence**: SQLite for local dev (`@cap-js/sqlite`), HANA Cloud in production (`@cap-js/hana`). Auth is `xsuaa` only under `[production]` profile (see `package.json` `cds.requires`); local dev runs unauthenticated. `xs-security.json` currently defines no scopes/role-templates — authorization is not yet modeled.

**Frontend** (`app/document-manager/`): a Fiori Elements List Report / Object Page app (SAPUI5 1.150, `sap.fe.templates`) generated with the Fiori Application Generator, backed by `mainService` = the `/api/documents/` OData service. Custom UI beyond the generated CRUD screens is added via the Object Page's `content.body.sections` extension in `manifest.json` (`ChatSection`), which loads a fragment + controller extension from `webapp/ext/fragment/`. Field/list annotations for `Documents` (labels, line items, facets) live in `app/document-manager/annotations.cds`, imported into `app/services.cds`.

**Sourcing frontend — Intake Hub built** (`app/intake-hub/`): first of the three §26 apps, mirroring `document-manager`'s exact structure/bootstrap (Fiori Elements v4, `sap.fe.templates`, UI5 1.150 via the SAPUI5 CDN) but against `/api/intake/`. Annotations live in `app/intake-hub/annotations.cds` (imported via `app/services.cds`). Two decisions made building it, both at the **service** (not domain-schema) layer to keep `db/sourcing-schema.cds` UI-annotation-free:
- `IntakeService.SourceDocuments` gained a computed `statusCriticality` column (`case status when 'FAILED' then 1 ... end`) so `UI.LineItem`/`UI.FieldGroup` can color-code status (red/yellow/green) via `Criticality: statusCriticality` — verified live in the browser (green ✓ EXTRACTED, red ✗ FAILED).
- `SourceDocuments` is now `@readonly` — Fiori Elements' auto-generated "Create" button would otherwise POST directly to the entity, bypassing `uploadDocument`'s validation (workspace existence, allowed `originType`). Creation only happens through the action; `@readonly` blocks raw CRUD (verified: direct POST → 405) without affecting the action handler's own internal `INSERT` (unaffected, since `@readonly` only gates the generic OData CRUD provider, not application code).

`extractRequirements`/`uploadDocument` are **unbound** actions (explicit `documentId`/`workspaceId` params), which don't map onto FE's per-row bound-action UI — rather than restructure those already-validated backend contracts, the Object Page gets a small custom section (`ext/fragment/IntakeActions.*`), reusing `ChatSection`'s exact `core:require`-fragment pattern, with a real "Extract Requirements" button calling the OData action via `ODataModel#bindContext(...).execute()`. Verified with a real headless-Chromium run (Playwright, since this environment has no `chromium-cli`): List Report renders with working filters and status colors, Object Page shows the annotated field group + custom Actions tab, and clicking the button flips a document from `UPLOADED` to `EXTRACTED` live, with the resulting `WorkspaceRequirement`/`RequirementSource` rows confirmed via OData afterward.

The Object Page also has a **"Document Content" section** (§26's raw extraction preview): a `UI.FieldGroup #Content` facet over `SourceDocument.content` with `@UI.MultiLineText`, placed between General Information and the custom Actions section (whose manifest anchor is `ContentFacet`). FE renders long text truncated with "Show More". Seeded demo documents predate the `content` field and show an empty preview; documents created via `uploadDocument` show their raw text.

**Upload flow — workspace picker + post-extract hand-off** (added after the initial build; the cross-app connective tissue from the intake→workspace plan): the List Report has a custom **"Upload Document"** toolbar action (manifest `controlConfiguration.@UI.LineItem.actions.uploadDocument` → `ext/fragment/UploadDialog`) opening a dialog that collects the **target workspace** (a `Select` over `RequirementWorkspaces` filtered to `status='OPEN'`), origin type, file name/type, and content, then calls `uploadDocument(workspaceId, …)`. This makes the "which workspace" routing a visible choice at upload instead of a buried parameter. To source that picker, `IntakeService` now exposes a **read-only `RequirementWorkspaces` projection** (creation still lives in the Requirement Workspace app; this is a picker source only). On the Object Page, `IntakeActions.onExtract` now ends a successful extraction with a **MessageBox offering "Open Workspace"** → deep-links to `/poc.sp.hub.requirementworkspace/index.html?workspace=<document.workspace_ID>` (mirrors the promote → "Open Project" hand-off). **Two FE-extension lessons** hit building the upload dialog (both found by headless driving): (1) a List Report `controlConfiguration` action handler is invoked with `this` = `sap.fe.templates.ListReport.ExtensionAPI` (which has `getModel`/`byId`/`loadFragment`/`refresh` but **no `getView`** and **no event argument**) — don't reach for `oEvent.getSource()`; (2) `ExtensionAPI.loadFragment` binds the fragment against the ExtensionAPI, which does **not** own custom dialog button handlers, so the fragment's `press="onConfirmUpload"` silently wires nothing — pass an explicit `controller:` object that owns the handlers, and address the fragment's children with `Fragment.byId(FRAGMENT_ID, localId)` (the ExtensionAPI's `byId` is view-scoped, not fragment-scoped).

**Change a document's workspace** (added later): `IntakeService.changeWorkspace(documentId, newWorkspaceId)` moves a `SourceDocument` to a different workspace, with a **block-after-extraction** rule — allowed while `UPLOADED`/`FAILED`, but **409 once `EXTRACTED`** ("… its requirements live in the current workspace. Delete them … first, then move"). Rationale (§18): an extracted document has already spawned `WorkspaceRequirement` rows in the old workspace; moving only the document would orphan them, so moving requirements too is deliberately out of scope for the PoC. Validates document + target workspace exist, rejects same-workspace and unknown-workspace, writes a `CHANGE_WORKSPACE` audit entry. UI: a **"Change Workspace"** button in the Object Page Actions section (`IntakeActions.onChangeWorkspace`) opens a workspace-picker dialog (`Select` over OPEN `RequirementWorkspaces`) → calls the action → refreshes. **FE binding lesson**: the button's declarative `enabled` binding (both `{= ${status} !== 'EXTRACTED'}` expression and a `formatter:`) is **not reliably applied** by FE for a custom Object-Page-section fragment control — the context resolves (`getProperty('status')` returns the value) but `enabled` stays true. So the block is enforced three ways: backend 409 (authoritative), a **handler pre-check** that shows an info MessageBox and skips the dialog when already EXTRACTED (the reliable UX gate), and the formatter (harmless if FE ever honors it). Same `#__select0`-style plain-`sap.m.Select` picker pattern as the upload dialog.

The **Requirement Workspace** app reads a `?workspace=<id>` deep-link in `onInit` (`_readWorkspaceParam`, tolerant of both search and hash query) and pre-selects that workspace over the OPEN-preferring default when it exists — this is the receiving end of the extract hand-off.

Registered like `document-manager`: `app/intake-hub/package.json` (npm workspace), root `package.json` `sapux` array + a `watch-intake-hub` script. `.prettierignore` now excludes all of `app/` (not just `document-manager`) to match the ESLint exclusion — every Fiori app has its own tooling/style.

**Sourcing frontend — Sourcing Project built** (`app/sourcing-project/`): second §26 app, same FE-v4 pattern against `/api/sourcing/`. List Report + Object Page with all six composition children (`Requirements`/`Risks`/`SourcingProjectSuppliers`/`SourcingProjectCommodities`/`Attachments` + General/Timeline field groups) as facets/tables. Three real lessons hit while building it (all found by driving the app headlessly, all fixed):

- A JSONModel declared in manifest `models` is **not reliably available** in the root view's `onInit` — create state models in `Component.init` *before* calling the parent init (which creates the root view).
- Don't rely solely on a binding's `dataReceived` to trigger initial UI state: on a warm server the data can arrive **before** the listener attaches. Check for already-present items first, then attach `change` + `dataReceived` with a run-once guard.
- `FilterBar#getConditions()` returns `{}` unless value persistence (p13n `Value` mode) is active — the table delegate reads `getInternalConditions()` (same shape, typed values) with `getConditions()` as fallback. The delegate maps conditions to `sap.ui.model.Filter`s manually (small operator map) instead of `FilterUtil` for version stability, and always ANDs in the `workspace_ID` scope filter from the `ui` model.
